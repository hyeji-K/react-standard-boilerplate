# 💡 React 공부 질문 리스트

리액트 프로젝트의 기본 구조를 공부하면서 했던 질문 정리

## 프로젝트 시작 프롬프트
```
"공부 목적으로 **표준적인 React 프로젝트(Vite 사용)**를 생성해줘.

단순히 파일만 만드는 게 아니라, 현업 프론트엔드 개발자들이 가장 많이 사용하는 폴더 구조(예: components, hooks, pages, services 등으로 분리)로 구성해줬으면 좋겠어.

그리고 각 폴더가 왜 필요한지, 보통 어떤 순서로 개발을 진행하는지 주석이나 README.md 파일에 간단히 설명해줘."
```

### npx create-vite@latest react-standard-example --template react
1. npx: 프로그램을 내 컴퓨터에 영구적으로 설치하지 않고, 필요할 때만 인터넷에서 잠깐 가져와서 실행하라는 도구입니다.
2. create-vite@latest: 'Vite'라는 최신 프로젝트 생성 도구를 사용하겠다는 뜻입니다. 예전에는 create-react-app을 많이 썼지만, 요즘 현업에서는 훨씬 빠른 Vite가 표준입니다.
3. react-standard-example: 생성될 폴더의 이름입니다.
4. --template react: "많은 프레임워크 중에 나는 React로 만들 거야"라고 지정해 주는 옵션입니다.

### npm install && npm install react-router-dom
1. npm install (기본 재료 준비)
- 설명: create-vite로 프로젝트를 만들었을 때, 설계도(package.json)는 생겼지만 실제 실행에 필요한 라이브러리 파일들은 아직 다운로드되지 않은 상태입니다.
- 역할: 설계도를 보고 React 실행에 필수적인 기본 파일들을 node_modules라는 폴더에 몽땅 다운로드합니다.
2. npm install react-router-dom (특수 부품 추가)
- 설명: React는 기본적으로 '한 페이지' 안에서 작동하는 도구입니다. 하지만 우리는 보통 "홈", "소개", "게시판"처럼 여러 페이지가 있는 웹사이트를 만들고 싶어 하죠.
- 역할: react-router-dom은 주소(URL)에 따라 다른 화면을 보여주는 '내비게이션(페이지 이동)' 기능을 담당하는 라이브러리입니다. 현업 프로젝트에서는 거의 100% 필수로 들어가는 라이브러리입니다.

### react-dom, react-router-dom
1. react-dom (화면 출력 담당)
리액트 코드를 실제 웹 브라우저의 눈에 보이게 **렌더링(출력)**해주는 필수 도구입니다.

- 역할: 리액트 컴포넌트(App.jsx 등)를 가져와서 HTML의 <div> 태그 안에 실제로 집어넣어 줍니다.
- 특징: 프로젝트를 만들면 기본으로 깔려 있습니다. 아까 본 main.jsx에서 ReactDOM.createRoot().render() 하는 부분이 바로 이 녀석을 쓰는 거예요.
- 비유: 영화 필름(리액트 코드)을 스크린(웹 브라우저)에 쏴주는 영사기 같은 존재입니다.

2. react-router-dom (길 찾기 담당)
웹사이트에서 **'주소(URL)에 따라 다른 화면'**을 보여주게 만드는 추가 도구입니다.

- 역할: 사용자가 /login 주소로 가면 로그인 페이지를, /home으로 가면 홈 페이지를 보여주는 기능을 합니다.
- 특징: 리액트 자체 기능이 아니라서, 아까처럼 npm install react-router-dom으로 직접 따로 설치해야 합니다.
- 비유: 극장에서 "1관은 액션", "2관은 코미디"라고 안내해주는 매표소 직원 같은 존재입니다.

### boilerplate 란?
- 프로그래밍에서는 **"매번 프로젝트를 시작할 때마다 반복적으로 작성해야 하는 기초적인 코드 뭉치"**를 의미
- npx create-vite가 바로 "React 보일러플레이트를 자동으로 생성해주는 도구"입니다.

```
💡 현업에서 보일러플레이트가 중요한 이유
- 속도: 기초 설정을 하는 데 시간을 낭비하지 않고 바로 기능을 만들 수 있습니다.
- 표준화: 팀원 모두가 똑같은 폴더 구조와 규칙을 쓰게 만들어주어 협업이 편해집니다.
- 품질: 이미 검증된 보안 설정이나 폴더 구조를 사용하므로 실수를 줄여줍니다.
```

### App.jsx랑 main.jsx 파일은 뭐고, 확장명은 왜 jsx야?
1. main.jsx vs App.jsx (역할 분담)
두 파일의 관계를 쉽게 비유하자면 **'전원 스위치'**와 **'첫 번째 가전제품'**의 관계입니다.
- main.jsx (입구/전원): * 우리 웹사이트가 시작되는 **가장 첫 번째 지점(Entry Point)**입니다. "HTML의 특정 구멍(id="root")을 찾아서, 거기에 우리 리액트 앱을 꽂아줘!"라고 명령하는 역할을 합니다. 보통 한 번 설정하면 자주 건드리지 않는 파일입니다.
- App.jsx (뿌리 컴포넌트): * 내 서비스의 실질적인 시작점입니다. main.jsx에 의해 호출되어 화면에 가장 먼저 그려집니다. 이 안에 내비게이션 바, 메인 화면, 푸터 등을 조립하게 됩니다. 모든 컴포넌트들의 '엄마'라고 생각하면 쉬워요.

2. 왜 확장자가 .jsx 인가요?
원래 자바스크립트는 .js를 쓰죠? 하지만 리액트에서는 **JSX(JavaScript XML)**라는 특별한 문법을 사용하기 때문에 .jsx를 씁니다. 
HTML과 JS의 결합: 자바스크립트 코드 안에서 HTML 태그(예: <div>, <h1>)를 바로 쓸 수 있게 해주는 문법입니다. 
시각적 편의: 코드를 딱 봤을 때 "아, 이건 화면 UI를 그리는 파일이구나!"라고 바로 알 수 있게 해줍니다.
도구의 도움: 에디터(VS Code 등)가 .jsx 확장자를 보면 "아, 리액트 코드니까 자동완성이랑 색깔 강조를 더 잘해줘야지!"라고 판단합니다.

### vite.config.js랑 eslint.config.js 파일은 머야?
1. vite.config.js (프로젝트 매니저)
이 파일은 프로젝트의 전반적인 운영 규칙을 정합니다.

- 역할: "이 프로젝트는 React를 사용해!", "서버는 5173번 포트에서 열어줘!" 같은 설정을 담당합니다.
- 왜 필요한가?: 브라우저는 사실 .jsx 파일을 바로 이해하지 못해요. Vite라는 매니저가 이 파일을 읽고 브라우저가 이해할 수 있는 일반 자바스크립트로 **번역(Build)**해주는데, 그 번역 방식이 이 파일에 적혀 있습니다.
- 언제 건드리나?: 나중에 절대 경로(예: ../../ 대신 @/ 쓰기)를 설정하거나, 외부 라이브러리를 연결할 때 아주 가끔 수정하게 됩니다.

2. eslint.config.js (코드 교관)
이 파일은 코드의 스타일과 문법 오류를 감시합니다.

- 역할: "너 변수 선언해놓고 왜 안 써?", "여기 세미콜론 빠졌어!"라고 잔소리를 해주는 역할입니다.
- 왜 필요한가?: 혼자 개발할 때는 귀찮을 수 있지만, 여러 명이 개발할 때 코드 스타일을 통일해주고, 아주 사소한 실수(오타 등)로 인해 프로그램이 멈추는 걸 미리 방지해 줍니다.
- 학습 포인트: 초보자일 때는 빨간 줄이 너무 많이 떠서 짜증 날 수도 있어요. 하지만 이 교관의 말을 잘 들으면 나중에 고생할 에러를 미리 잡을 수 있습니다.

```
💡 정리하자면
- App.jsx: 내가 직접 요리하는 '음식(코드)'
- vite.config.js: 요리를 도와주는 '가스레인지와 오븐 설정(도구 설정)'
- eslint.config.js: 위생 상태와 조리법을 체크하는 '위생 검사관(코드 검사)'
```

### index.html
**index.html**은 우리 리액트 앱의 **'도화지'**이자 **'유일한 실제 페이지'**입니다.

리액트는 SPA (Single Page Application) 방식이라고 들어보셨나요? 페이지는 딱 이 index.html 하나뿐이고, 그 안의 내용물만 리액트가 슥슥 바꿔치기하며 여러 페이지처럼 보여주는 거예요.

1. 파일 안을 들여다보면 (핵심 포인트)
index.html 파일을 열어보면 딱 두 군데가 제일 중요합니다.

```HTML
<div id="root"></div>

<script type="module" src="/src/main.jsx"></script>
```

- <div id="root"></div>: 여기가 바로 리액트 앱이 들어갈 자리입니다. 처음엔 텅 비어있지만, 앱이 실행되면 리액트가 이 구멍을 찾아서 우리가 만든 컴포넌트들을 이 안에 쏟아붓습니다.
- <script src="/src/main.jsx">: 아까 공부한 main.jsx(전원 스위치)를 불러오는 줄입니다. 이 줄이 있어야 리액트 엔진이 가동됩니다.

2. 왜 여기엔 코드를 많이 안 짜나요?
리액트 개발을 할 때는 이 파일을 거의 건드리지 않습니다. 왜냐하면:

주인공은 리액트: 화면에 보이는 모든 것은 .jsx 파일에서 만들기 때문이죠.

설정용: 웹사이트의 제목(<title>), 파비콘(아이콘), 폰트 설정 같은 기본 배경 설정을 할 때만 아주 가끔 열어봅니다.

3. 전체 흐름 (한눈에 보기)
지금까지 배운 파일들의 연결 고리를 완성해 드릴게요.

사용자가 웹사이트 접속 → **index.html**이 로드됨.

index.html이 **main.jsx**를 호출함.

main.jsx가 "너 root라는 구멍 찾아서 거기에 App.jsx 그려!"라고 명령함.

드디어 우리 눈에 화면이 보임!

### public 폴더
public 폴더는 한마디로 **"리액트가 건드리지 않는 정적 자원(Static Assets) 저장소"**입니다.
보통 우리가 만드는 .jsx 파일들은 Vite라는 도구가 코드를 분석하고 압축하는 과정을 거치지만, public 폴더에 있는 파일들은 "원본 그대로" 웹사이트에 배포됩니다.

1. public vs src/assets (제일 헷갈리는 부분)
프로젝트를 보면 src 안에도 assets 폴더가 있고, 밖에도 public이 있어서 헷갈리실 거예요. 이 둘의 차이점만 알아도 중급자 소리를 듣습니다!

2. public 폴더의 특징주소가 정직합니다: public/photo.jpg라고 저장했다면, 웹사이트 주소 뒤에 바로 /photo.jpg를 붙여서 확인할 수 있습니다.
- 리액트 외부 자원: 리액트 엔진이 이 파일을 읽어서 해석하지 않습니다. 그냥 "이건 나중에 서버에 그대로 올려줘"라고 맡겨두는 바구니 같은 곳이에요.
- index.html에서 직접 접근: index.html에서 <link> 태그로 폰트나 아이콘을 불러올 때 주로 사용합니다.

3. 언제 public을 써야 할까요?
- 파비콘(Favicon): 브라우저 탭에 뜨는 작은 아이콘.
- robots.txt: 구글 같은 검색 엔진 로봇에게 "우리 사이트 어디까지 검색해도 돼"라고 알려주는 설정 파일.
- 대용량 파일: 너무 커서 리액트가 처리하기 버거운 동영상이나 PDF 파일 등.
- 파일 이름이 바뀌면 안 될 때: src/assets에 넣으면 보안과 성능을 위해 파일 이름 뒤에 이상한 문자열(해시)이 붙지만, public은 이름을 그대로 유지합니다.

### index.css
**index.css**는 우리 앱의 **'전체적인 스타일 가이드라인'**을 결정하는 파일입니다.

리액트에는 스타일을 입히는 방법이 여러 가지가 있는데, 그중에서도 index.css는 가장 넓은 범위에 영향을 미치는 파일이라고 생각하면 됩니다.

1. index.css vs App.css (범위의 차이)
보통 프로젝트를 만들면 CSS 파일이 두 개 보일 텐데, 그 차이는 **"누구에게 옷을 입히느냐"**에 있습니다.

index.css (전체 유니폼):

**전역 스타일(Global Style)**을 담당합니다.

웹사이트 전체의 배경색, 기본 폰트, 여백 초기화(Reset) 등 모든 페이지와 모든 컴포넌트에 공통으로 적용될 스타일을 적습니다.

main.jsx에서 임포트(import)되어 앱 전체에 뿌려집니다.

App.css (특정 옷):

**지역 스타일(Local Style)**을 담당합니다.

오직 App.jsx라는 특정 컴포넌트 안에서만 쓰이는 스타일을 적습니다.

2. 왜 index.css를 따로 쓰나요?
브라우저마다 기본적으로 설정된 여백이나 글꼴이 조금씩 다릅니다. 이걸 그대로 두면 크롬에서 볼 때와 사파리에서 볼 때 화면이 미묘하게 달라질 수 있죠.

그래서 index.css에 다음과 같은 설정을 주로 넣습니다.

- CSS Reset: 모든 브라우저의 기본 여백을 0으로 초기화하기.

- 공통 폰트 설정: 사이트 전체에서 쓸 예쁜 폰트 지정하기.

- 박스 모델 설정: 테두리(border)를 포함해서 크기를 계산하도록 설정(box-sizing: border-box).
